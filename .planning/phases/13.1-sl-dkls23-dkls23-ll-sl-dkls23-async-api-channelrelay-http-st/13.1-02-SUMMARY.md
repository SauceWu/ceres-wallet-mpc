# Phase 13.1 Plan 02: mpc_engine.rs Rewrite for sl-dkls23 Async API Summary

**One-liner:** Replaced all dkls23-ll 4-round state machines with sl-dkls23 async run() calls bridged via ChannelRelayConn channels, and swapped manual Lagrange key export for combine_shares.

## What Was Built

### Task 1: Rewrite keygen/sign/recover

**Architecture pattern (all three protocols follow the same shape):**

1. Parse WireEnvelope, validate `from_id==1`
2. Base64-decode `server_env.payload` to get raw protocol bytes
3. Build `SetupMessage` (2-of-2, party_id=0, NoSigningKey, NoVerifyingKey::new(0/1))
4. Create mpsc channel pair: `(tx_in, rx_in)` + `(tx_out_unbounded, rx_out_unbounded)`
5. Construct `ChannelRelayConn { rx: rx_in, tx: tx_out_unbounded }`
6. `get_runtime().spawn(async move { protocol::run(setup, seed, relay).await })`
7. `get_runtime().block_on(tx_in.send(server_msg_bytes))` — inject first server message
8. `get_runtime().block_on(rx_out_unbounded.recv())` — read first client message
9. Store `{ tx_in, rx_out, task_handle }` in session map
10. Return `MpcRoundResult(in_progress)` with client message wrapped in WireEnvelope

**Continue pattern:**
- Clone `tx_in` from session, inject server message
- `block_on(session.rx_out.recv())`:
  - `Some(bytes)` → wrap in WireEnvelope, return in_progress
  - `None` → channel closed (protocol done) → await task_handle for final result

**Protocol-specific setup:**
- **keygen**: `sl_dkls23::keygen::dkg::run` → `Result<Keyshare>` → `as_slice()` → Base64
- **sign**: `sl_dkls23::sign::run` with `Arc<Keyshare>` + `with_hash(digest)` → `(sig_bytes, recid)` tuple
- **recover**: `key_refresh::run` with `KeyshareForRefresh::from_keyshare(&old_ks, None)` → new Keyshare → Base64

**Security guards preserved:**
- T-13.1-04: EXPORTED_KEYS checked in `sign_start` before spawn
- SEC-01: `consumed` flag set to true before `rx_out.recv()` in `sign_continue`; reset to false if intermediate message received
- SEC-02: `SESSION_TTL` check with single lock scope in `recover_continue`
- from_id==1 validated on every WireEnvelope

### Task 2: Backup adaptation + Key export rewrite

**Backup functions (no structural change needed):**
- `derive_backup_envelope`: encrypts `local_encrypted_share.as_bytes()` via AES-256-GCM+HKDF — works with both old JSON and new Base64 strings transparently
- `decrypt_backup_share`: decrypts and returns the stored string (now Base64 keyshare bytes)

**Key export (`export_private_key`):**
- Deleted `KeyshareExportFields` intermediate struct (no longer needed)
- Deleted `lagrange_coeff_2of2` helper function
- Replaced with `sl_dkls23::key_export::combine_shares(&x_i_combined, &s_i_combined, &pk)`
- Keyshare internals accessed via public methods: `x_i_list()`, `rank_list()`, `s_i()`, `public_key()`
- `combine_shares` internally verifies `G * sk == pk` (T-13.1-09 mitigated)
- EXPORTED_KEYS registration still present (T-13.1-04)

**Dead code removed:**
- `KeyshareExportFields` struct
- `lagrange_coeff_2of2` function
- All `dkls23_ll::*` imports
- `encode_cbor_base64` / `decode_cbor_base64` helpers (no longer needed)

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] SignSetup::new takes Arc<Keyshare>, not ranks/t parameters**
- **Found during:** Task 1 sign implementation
- **Issue:** RESEARCH.md documented `SignSetup::new(inst, sk, party_id, vk, ranks, t)` — but the actual sl-dkls23 1.0.0-beta `setup::sign::SetupMessage::new` takes `(instance, sk, party_idx, vk, share: Arc<KS>)`. There are no ranks/t parameters; those are embedded in the Keyshare.
- **Fix:** Used `Arc::new(keyshare)` and passed to `SignSetup::new(inst, NoSigningKey, 0, vk, keyshare_arc)`.
- **Files modified:** rust/src/api/mpc_engine.rs
- **Commit:** 730732c

**2. [Rule 1 - Bug] NoVerifyingKey::new(id) requires party ID argument**
- **Found during:** Task 1 keygen/sign implementation
- **Issue:** RESEARCH.md showed `vec![NoVerifyingKey; 2]` (default construction). Actual API: `NoVerifyingKey::new(id: usize)` — requires a party ID for identity.
- **Fix:** Used `vec![NoVerifyingKey::new(0), NoVerifyingKey::new(1)]` for 2-of-2.
- **Files modified:** rust/src/api/mpc_engine.rs
- **Commit:** 730732c

**3. [Rule 1 - Bug] sign::dsg module is private — use sl_dkls23::sign::run directly**
- **Found during:** Task 1 compilation
- **Issue:** Called `sign::dsg::run(...)` but `dsg` is a private module. The sign module re-exports via `pub use dsg::*`.
- **Fix:** Changed to `sl_dkls23::sign::run(setup, seed, relay)`.
- **Files modified:** rust/src/api/mpc_engine.rs
- **Commit:** 730732c

**4. [Rule 1 - Bug] setup::keys module is private — use sl_dkls23::setup::{NoSigningKey, NoVerifyingKey}**
- **Found during:** Task 1 compilation
- **Issue:** `sl_dkls23::setup::keys` is `mod keys` (private), but the parent `setup` re-exports via `pub use keys::*`.
- **Fix:** Changed import to `use sl_dkls23::setup::{NoSigningKey, NoVerifyingKey}`.
- **Files modified:** rust/src/api/mpc_engine.rs
- **Commit:** 730732c

**5. [Rule 3 - Blocking] frb_generated.rs MessageDigest private field access**
- **Found during:** cargo check
- **Issue:** frb_generated.rs used `MessageDigest(var_field0)`, `self.0`, and `self.0` to access the private tuple field. These were pre-existing issues that now blocked compilation after mpc_engine.rs was cleaned up.
- **Fix:** Replaced with `MessageDigest::new(var_field0)`, `self.into_bytes()`, and `self.into_bytes()`.
- **Files modified:** rust/src/frb_generated.rs
- **Commit:** 730732c

**6. [Rule 1 - Bug] SignSession.task_handle type is Result<(Vec<u8>, u8), String>**
- **Found during:** Task 1 compilation
- **Issue:** Initial implementation packed r+s+recid into a single `Vec<u8>`, returning `Result<Vec<u8>, String>`. But `SignSession.task_handle` type is `JoinHandle<Result<(Vec<u8>, u8), String>>`.
- **Fix:** Changed spawn closure to return `(sig_bytes, recid.to_byte())` tuple; updated sign_continue to destructure `(sig_bytes, recid)`.
- **Files modified:** rust/src/api/mpc_engine.rs
- **Commit:** 730732c
