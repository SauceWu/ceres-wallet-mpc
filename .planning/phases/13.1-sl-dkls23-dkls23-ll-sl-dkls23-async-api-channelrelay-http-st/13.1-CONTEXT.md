# Phase 13.1: sl-dkls23 迁移 - Context

**Gathered:** 2026-04-09
**Status:** Ready for planning

<domain>
## Phase Boundary

将 dkls23-ll 低层同步 API 替换为 sl-dkls23 高层 async API。引入 ChannelRelay 桥接 HTTP start/continue FFI 模式。适配 Keyshare 二进制序列化格式。使用内置 key_export::combine_shares 替换手动 Lagrange 插值，使用 key_refresh 替换手动 rotation 实现。验证 iOS + Android 双平台编译通过。

</domain>

<decisions>
## Implementation Decisions

### ChannelRelay 桥接架构
- 实现自定义 Relay trait，内部用 mpsc channel 对接 FFI start/continue 调用
- 使用 OnceCell 懒初始化单例 tokio current_thread runtime，进程级别不主动销毁
- keygen_start/sign_start 创建 channel pair 并 spawn async protocol task
- keygen_continue/sign_continue 通过 channel 推入 server 消息，等待 client 输出
- 保留 FFI 层 start/continue 模式不变，Dart 层零改动
- 服务端是套壳实现（非直接 dkls23），Relay 实现负责 HTTP 消息转发兼容

### Keyshare 序列化格式
- 使用 sl-dkls23 原生 bytes 序列化，存储为 Base64 字符串
- 不兼容旧 dkls23-ll JSON Keyshare 格式，不做迁移（全面切换）
- backup envelope 直接接收 Keyshare 序列化后的 bytes

### 编译验证与测试
- 本地 cargo check --target aarch64-apple-ios + aarch64-linux-android 验证
- CI 门控（ci.yml）自动交叉编译验证
- 保留现有 Rust 双方模拟测试框架，改用 sl-dkls23 API 重写
- 协议测试用 #[tokio::test] 替换 #[test]
- key_export/combine_shares 是 sync 的，保持简单单元测试
- key_refresh 复用 Phase 12 测试模式

### Claude's Discretion
- ChannelRelay 内部 channel 类型选择（tokio::sync::mpsc vs crossbeam）
- WireEnvelope 格式是否需要调整以适配 sl-dkls23 消息格式
- session.rs 结构体重构细节
- FRB codegen 重新生成时机和步骤

</decisions>

<specifics>
## Specific Ideas

- sl-dkls23 API: keygen::dkg::run(setup, seed, relay) → Result<Keyshare>
- sl-dkls23 API: sign::dsg::run(setup, seed, relay) → Result<(Signature, RecoveryId)>
- sl-dkls23 API: key_export::combine_shares(x_i_list, s_i_list, public_key) → Option<Scalar> (sync)
- Relay trait 来自 sl_mpc_mate::coord，需要自定义实现
- SimpleMessageRelay 示例在 dkls23 examples/ 中可参考

</specifics>
