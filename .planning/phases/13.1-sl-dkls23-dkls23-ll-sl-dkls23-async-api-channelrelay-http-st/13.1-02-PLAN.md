Rewrite the entire mpc_engine.rs file. The new architecture for keygen/sign/recover follows this pattern:

**`xxx_start(session_id, ..., server_payload)` pattern:**
1. Parse server_payload WireEnvelope JSON, validate from_id == 1
2. Extract opaque payload bytes from WireEnvelope (these are the raw protocol bytes to inject into the channel)
3. Create channel pair: `(tx_in, rx_in)` for FFI->protocol, `(tx_out_unbounded, rx_out_unbounded)` for protocol->FFI
4. Construct `ChannelRelayConn { rx: rx_in, tx: tx_out_unbounded }`
5. Build SetupMessage (2-of-2, party_id=0, NoSigningKey/NoVerifyingKey, ranks=[0,0], t=2, InstanceId from session_id hex)
6. Generate random seed: `[u8; 32]`
7. `get_runtime().spawn(async move { keygen::run(setup, seed, relay).await })` — protocol task runs independently
8. Inject server's first message bytes into the channel: `get_runtime().block_on(tx_in.send(server_msg_bytes))`
9. Read first client output: `get_runtime().block_on(rx_out_unbounded.recv())` — this is the client's first protocol message
10. Store session `{ tx_in, rx_out: rx_out_unbounded, task_handle }` in global KEYGEN_SESSIONS
11. Wrap client output bytes in WireEnvelope, return MpcRoundResult(status="in_progress")

**`xxx_continue(session_id, server_payload)` pattern:**
1. Parse server_payload WireEnvelope JSON, validate from_id == 1
2. Extract opaque payload bytes
3. Get session from global map (lock, get_mut, unlock)
4. Inject server message: `get_runtime().block_on(session.tx_in.send(msg_bytes))`
5. Read next client output: `get_runtime().block_on(session.rx_out.recv())`
6. If `recv()` returns None AND task_handle is available → protocol completed, `.await` the JoinHandle to get final result
7. Protocol result:
   - Keygen: `Result<Keyshare>` → `keyshare.as_slice()` → Base64 encode → `local_encrypted_share`
   - Sign: `Result<(Signature, RecoveryId)>` → extract r, s, recid
   - Recovery: `Result<Keyshare>` → same as keygen completion
8. If `recv()` returns Some(bytes) → intermediate message, wrap in WireEnvelope, return MpcRoundResult(status="in_progress")

**CRITICAL ARCHITECTURE DETAIL — Message format:**
The WireEnvelope.payload field currently contains CBOR-base64 encoded dkls23-ll typed messages. With sl-dkls23, the Relay trait exchanges raw opaque `Vec<u8>` bytes. The question is: what format does the server send/expect?

Since the server is a "wrapper implementation" (per CONTEXT.md: "服务端是套壳实现"), the plan MUST:
- Decode WireEnvelope.payload from Base64 to get raw bytes
- Pass those raw bytes directly to the channel (the sl-dkls23 protocol task handles parsing)
- When reading from rx_out, Base64-encode the raw bytes and put into WireEnvelope.payload

This means WireEnvelope is preserved as the HTTP transport envelope, but its payload is now opaque sl-dkls23 protocol bytes (Base64-encoded), NOT CBOR-encoded typed messages. The `encode_cbor_base64`/`decode_cbor_base64` helpers for protocol messages are NO LONGER NEEDED (though they may still be used for WireEnvelope JSON itself or completed payloads).

**CRITICAL: round tracking.**
The old code tracked rounds explicitly (round 2, 3, 4). With sl-dkls23 async, the protocol handles rounds internally. The FFI layer simply:
- Pushes server messages → gets client messages (via channel)
- Until the channel closes (protocol done) → read final result from JoinHandle

So `MpcRoundResult.round` can be a simple counter incremented on each continue call, or use `WireEnvelope.round` from server message. The exact round semantics are now cosmetic.

**For keygen_start/keygen_continue:**
- Use `sl_dkls23::keygen::run(setup, seed, relay)` which returns `Result<Keyshare, KeygenError>`
- On completion: `keyshare.as_slice()` → Base64 → `local_encrypted_share`
- EVM address: `keyshare.public_key()` → `to_affine()` → `to_encoded_point(false)` → `derive_evm_address()`

**For sign_start/sign_continue:**
- Use `sl_dkls23::sign::run(setup, seed, relay)` which returns `Result<(Signature, RecoveryId), SignError>`
- `sign_start` args remain: `(session_id, share, message_hash_hex, server_payload)`
- `share` is now Base64 keyshare bytes (not JSON) — `Keyshare::from_bytes(&BASE64_STANDARD.decode(&share)?)`
- SetupMessage: `SignSetup::new(...).with_hash(digest.into_bytes()).with_chain_path("m".parse()?)`
- EXPORTED_KEYS guard: extract public_key compressed hex before spawning task, check against EXPORTED_KEYS
- On completion: extract r, s bytes from Signature, recid from RecoveryId

**For recover_start/recover_continue:**
- Use `sl_dkls23::keygen::key_refresh::run(setup, seed, relay, share_for_refresh)`
- `backup_share` is now Base64 keyshare bytes — `Keyshare::from_bytes(&BASE64_STANDARD.decode(&backup_share)?)`

- On completion: `new_keyshare.as_slice()` → Base64 → `local_encrypted_share`

**Keep the existing function signatures** — the pub fn signatures for keygen_start, keygen_continue, sign_start, sign_continue, recover_start, recover_continue MUST remain identical (same parameter names and types) to preserve FRB compatibility.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| FFI payload → channel | Server WireEnvelope payload decoded and injected into protocol channel |
| channel → FFI response | Protocol output bytes wrapped in WireEnvelope for HTTP transport |
| Keyshare bytes → export | Two keyshare byte arrays combined to reconstruct private key |
