<threat_model>
## Phase 1 安全威胁模型

1. **Stub 数据泄漏风险** — LOW
   - 威胁: Stub 实现可能返回看似真实的密钥材料
   - 缓解: 所有 stub 返回值必须以 "stub_" 为前缀，明确标识为假数据
   - 验证: grep stub 返回值确认无 real-looking secret

2. **Rust 依赖链审计** — LOW
   - 威胁: 引入不可信的 Rust crate
   - 缓解: Phase 1 仅依赖 flutter_rust_bridge + serde + serde_json，均为 Rust 生态广泛审计的标准库；不引入 kms-secp256k1 或任何密码学依赖
   - 验证: cargo tree 不包含密码学库

3. **FRB Codegen 暴露内部类型** — LOW
   - 威胁: FRB codegen 可能将 Rust 内部类型暴露给 Dart 侧
   - 缓解: 仅在 rust/src/api/ 模块中使用 pub fn，内部类型（如 MpcRoundResult）仅用于 serde 序列化，不作为函数参数/返回类型暴露
   - 验证: FRB 生成的 Dart 文件中只包含 String 参数的函数签名

4. **FFI 边界内存安全** — LOW
   - 威胁: 手动 FFI 可能导致内存泄漏或缓冲区溢出
   - 缓解: 使用 FRB v2 全自动内存管理，FFI 边界仅传递 String（JSON），不手写 dart:ffi
   - 验证: 无手动 dart:ffi import
</threat_model>

    **Step 3: 创建 rust/src/api/mpc_engine.rs**
    实现 6 个 stub 函数（per D-08），函数签名严格如下:
    - `pub fn keygen_start(session_id: String, server_payload: String) -> Result<String, String>`
    - `pub fn keygen_continue(session_id: String, server_payload: String) -> Result<String, String>`
    - `pub fn recover_start(session_id: String, backup_share: String, server_payload: String) -> Result<String, String>`
    - `pub fn recover_continue(session_id: String, server_payload: String) -> Result<String, String>`
    - `pub fn sign_start(session_id: String, share: String, server_payload: String) -> Result<String, String>`
    - `pub fn sign_continue(session_id: String, server_payload: String) -> Result<String, String>`

    每个函数:
    1. 使用 `serde_json::from_str` 验证 server_payload 是有效 JSON（或直接接受 String）
    2. 构造 `MpcRoundResult` 并 `serde_json::to_string` 序列化为 JSON 返回
    3. stub 返回值必须用 "stub_" 前缀标记（安全威胁缓解）

    keygen_start 返回示例:
    ```json
    {"status":"continue","round":1,"client_payload":"stub_keygen_round1_{session_id}","error_message":null}
    ```
    keygen_continue 返回示例:
    ```json
    {"status":"completed","round":2,"client_payload":"stub_keygen_completed_{session_id}","error_message":null}
    ```
    recover/sign 函数模式相同，使用对应前缀 "stub_recover_"/"stub_sign_"。
