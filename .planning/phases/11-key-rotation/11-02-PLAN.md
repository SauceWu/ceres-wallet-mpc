**Test 3: test_rotation_version_increments (PROTO-03)**
- This tests the API layer: create an in-process server simulation
- DKG -> get share0 (device), share1 (server)
- Serialize share0 as backup_share JSON
- Simulate server Round 1: create server's rotation state via `DkgState::key_rotation(&share1, &mut rng)`, generate_msg1, wrap in WireEnvelope JSON with protocol=rotation
- Call `ceres_mpc::api::mpc_engine::recover_start(session_id, backup_share_json, server_env_json, 5)` where 5 is the current rotation version
- Continue driving recover_continue + server side through 4 rounds, passing WireEnvelope JSON strings
- Parse final MpcRoundResult, extract RecoveryCompletedPayload, assert `rotation_version == 6` (5 + 1)

NOTE: This test is complex (API layer with WireEnvelope serialization). If it proves too complex within context budget, simplify to a unit-level test that directly tests the version increment logic by inserting a RecoverySession with current_rotation_version=5 into RECOVERY_SESSIONS and simulating only Round 4 completion. The KEY assertion is `rotation_version == current + 1`.

- Create a dummy RecoverySession with `created_at: Instant::now() - (SESSION_TTL + Duration::from_secs(1))`

  - Set round=2, current_rotation_version=1, all Option fields None

- Call `ceres_mpc::api::mpc_engine::recover_continue("test_ttl_session".to_string(), dummy_server_envelope_json)`

For the `created_at` backdating, use: `Instant::now() - (SESSION_TTL + Duration::from_secs(1))`
