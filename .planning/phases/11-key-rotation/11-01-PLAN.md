**recover_start implementation** (mirrors keygen_start but uses State::key_rotation):

1. Parse server_payload as WireEnvelope, validate `from_id == 1`
2. Deserialize `backup_share` as `dkls23_ll::dkg::Keyshare` via `serde_json::from_str` — this is the decrypted backup share, NOT a BackupEnvelope
3. Call `DkgState::key_rotation(&old_keyshare, &mut rng).map_err(|e| e.to_string())?` — CRITICAL: returns Result, must unwrap
4. Call `state.generate_msg1()` (discard result, same as keygen_start)
5. Decode server_msg1 from envelope payload via `decode_cbor_base64::<dkls23_ll::dkg::KeygenMsg1>`
6. Call `state.handle_msg1(&mut rng, vec![server_msg1])` to get Vec<KeygenMsg2>
7. Encode msg2[0] as WireEnvelope with `ProtocolType::Rotation`, round=2, from_id=0, to_id=Some(1)
8. Insert RecoverySession into RECOVERY_SESSIONS with `created_at: Instant::now()`, `round: 2`, `current_rotation_version`
9. Return MpcRoundResult { status: "in_progress", round: 2, client_payload: env_json }

**recover_continue implementation** (mirrors keygen_continue with TTL check):

At entry, BEFORE any round logic, perform TTL check within a single lock scope:
```rust
let current_round = {
    let mut sessions = RECOVERY_SESSIONS.lock().unwrap();
    match sessions.get(&session_id) {
        None => return Err(format!("recovery session not found: {session_id}")),
        Some(s) if s.created_at.elapsed() > SESSION_TTL => {
            sessions.remove(&session_id);
            return Err(format!("session expired (TTL): {session_id}"));
        }
        Some(s) => s.round,
    }
};
```

Then match on current_round with EXACTLY the same arms as keygen_continue:
- Round 2: handle server KeygenMsg2 -> calculate_commitment_2 -> return commitment envelope (step="commitment", protocol=Rotation)
- Round 3 (step="commitment"): cache server_commitment_2, return pending_msg3 envelope (step="msg3", protocol=Rotation)  
- Round 3 (step="msg3"): handle server KeygenMsg3 with commitment_2_list -> return KeygenMsg4 envelope (protocol=Rotation)
- Round 4: handle server KeygenMsg4 -> get new Keyshare -> derive EVM address -> build RecoveryCompletedPayload with `rotation_version: session.current_rotation_version + 1`

All WireEnvelope construction uses `ProtocolType::Rotation` (not Dkg).

CRITICAL anti-patterns to avoid (from RESEARCH.md):
- Do NOT call `finish_key_rotation` — it does not exist in locked version c348be1
- Do NOT forget to unwrap the Result from `State::key_rotation`
- TTL check and session.remove MUST happen in the same lock() scope
- `current_rotation_version` is stored in session, only used in Round 4 completion

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| server -> client (WireEnvelope) | Server envelope payload may be malicious; dkls23-ll protocol layer provides cryptographic verification |
| backup_share input | JSON string from Dart layer; must be valid Keyshare JSON (post-decrypt) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-11-01 | Spoofing | recover_continue WireEnvelope from_id | mitigate | validate `from_id == 1` at recover_continue entry (inherited from keygen_continue pattern) |
| T-11-02 | Tampering | WireEnvelope payload | accept | dkls23-ll protocol layer provides cryptographic binding (commitment_2 + hash verification); not our layer to re-verify |
| T-11-03 | Denial of Service | RECOVERY_SESSIONS memory exhaustion from stale sessions | mitigate | SEC-02: SESSION_TTL=300s with lazy eviction at recover_continue entry; expired sessions removed in same lock scope |
| T-11-04 | Elevation of Privilege | Reuse of old Keyshare after rotation | accept | Keyshare implements ZeroizeOnDrop (verified c348be1 L200); old share s_i replaced by protocol — old share signatures fail cryptographic verification |
| T-11-05 | Information Disclosure | rotation_version leaks rotation count | accept | rotation_version is non-sensitive metadata; no PII involved |
| T-11-06 | Spoofing | Forged session_id in recover_continue | accept | session_id is 32-byte random hex generated client-side; collision probability negligible |
